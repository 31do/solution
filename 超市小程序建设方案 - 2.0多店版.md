# 超市小程序-多店版开发解决方案

**环企网络 | 2026年4月**

---

## 一、提出痛点

当前拥有多家分店的连锁超市在数字化转型中普遍面临以下核心痛点：

- **获客渠道单一**：过度依赖线下自然客流和促销传单，缺乏线上精准引流与裂变机制，新客获取成本高且不可持续。
- **门店管理松散**：各分店商品、库存、订单、配送独立运营，总部无法实时掌握各店经营数据，难以统一调配资源，造成部分门店商品积压、另一部分缺货。
- **客户忠诚度低**：传统会员卡使用率低，无法记录消费行为，难以开展个性化营销。老客户流失严重，复购率不足30%。
- **配送效率低下**：大部分超市仍依赖第三方外卖平台，抽成高达15%-20%，且无法自主控制配送时效与服务体验。顾客投诉“送错门店”“配送超时”频发。
- **技术能力薄弱**：自建线上系统投入大、周期长，依赖通用SaaS平台又受限于功能模板，无法满足连锁超市特有的“就近门店分配”“多店独立库存”“预约配送”等复杂业务场景。

## 二、解决方案

基于微信小程序生态，为连锁超市量身打造一套“多店版O2O零售系统”，实现以下核心能力：

- **线上线下一体化**：消费者可通过小程序浏览商品、下单、支付，订单自动分配到最近可配送门店；支持到店自提与预约配送。
- **多门店独立运营**：每家分店拥有独立后台管理商品、库存、订单、配送员；总部可查看所有门店数据并统一配置营销活动。
- **智能化会员营销**：集成等级会员、积分商城、优惠券、秒杀、拼团、满减等工具，系统自动分析消费行为并推送个性化优惠。
- **自主配送管理体系**：提供店长指派或智能派单模式，配送员端支持路径导航、签收拍照、历史轨迹留存，完全掌控配送成本与服务质量。
- **数据驱动决策**：实时生成各门店销售报表、商品畅销榜、用户复购分析，为补货、选品、促销提供精准依据。

## 三、业务需求

1. **小程序用户端**：支持微信授权登录、商品多级分类展示、搜索、购物车、在线支付、订单跟踪、会员中心、积分兑换、秒杀/拼团/优惠券等营销参与。
2. **管理后台（PC网页）**：
   - 超级管理员：门店管理、商品全局管理、营销活动创建、数据看板、权限分配。
   - 门店店长：本店商品上下架、库存调整、订单处理、配送指派、本店数据报表。
   - 配送员：接收派单、扫码取货、开始配送、完成送达（拍照）。
3. **多门店智能分配引擎**：根据用户LBS自动匹配3公里内可配送门店，若超出范围则引导到店自提。
4. **数据安全与合规**：全链路SSL加密，支持小程序备案代办，每日自动备份。
5. **高扩展性**：后续可平滑增加生鲜配送、批发商城、社区团购等模块。

## 六、用户端功能与栏目（微信小程序）

### 6.1 首页与全局导航

**应用场景**：用户首次进入小程序，需要快速了解超市促销活动、热门商品，并导航至分类、购物车、个人中心。

**实施分析**：首页采用组件化设计，后台可动态配置轮播图、金刚区图标（限时秒杀、拼团、积分商城等）、促销橱窗（节日专题）。底部Tab栏固定“首页、分类、购物车、我的”。

**实现技术/方法**：前端Vue动态渲染，后端提供配置接口（/api/home/config）。缓存Redis存储首页配置，减少数据库查询。利用 **旺道UI框架WDUI** 构建响应式网格布局，确保不同手机屏幕适配。

**算法与数据流**：
- 用户进入首页 → 请求配置接口 → 后端从Redis读取JSON配置 → 若无则从数据库读取并写入Redis → 返回前端渲染。
- 商品推荐位：基于用户历史浏览（Redis记录最近5个商品ID）和热门销量（PostgreSQL每日聚合销量前20）混合排序，加权公式：`score = 0.6*近7日销量 + 0.4*用户偏好匹配度`。

**操作流程**：下拉小程序 → 看到轮播图展示本周特价水果 → 点击金刚区“秒杀”进入活动页 → 底部Tab切换。

**FAQ**：
- Q：轮播图多久可以更换一次？A：后台管理员可随时上传新图，设置展示时间范围，即时生效。
- Q：首页加载慢怎么办？A：后台已配置CDN加速图片，接口启用Redis缓存，正常打开速度<1秒。

### 6.2 商品分类与搜索

**应用场景**：用户想买饮料，通过分类树逐级找到碳酸饮料；或直接搜索“可乐”，按价格、销量排序。

**实施分析**：商品分类支持三级（一级：食品饮料；二级：饮料；三级：碳酸饮料）。搜索支持模糊匹配、拼音首字母（“kel”找到可乐）。结果可按综合、销量、价格排序。

**实现技术/方法**：PostgreSQL存储分类树（parent_id递归查询）；搜索采用`ilike`全库扫描（数据量<5万时性能可接受），未来可升级Elasticsearch。排序使用SQL order by。

**算法与数据流**：
- 分类页面：请求分类ID → 后端查询该分类下所有商品（包含子分类，递归查询postgres递归CTE）→ 分页返回。
- 搜索流程：用户输入关键词 → 前端防抖500ms → 后端拼接`name ilike '%关键词%'` → 按排序字段返回。

**操作流程**：点击底部“分类” → 看到食品、日化、生鲜三个一级 → 点击“食品”展开二级 → 点击“饮料”展开三级“碳酸饮料/果汁” → 进入商品列表。

**FAQ**：
- Q：会不会出现重复分类？A：后台分类名称唯一校验，且支持拖拽排序。
- Q：能否根据用户所在门店展示不同分类顺序？A：支持，后台可为每个门店独立配置分类展示顺序（扩展表store_category_order）。

### 6.3 商品详情页

**应用场景**：用户点击某个商品，查看大图、价格、库存、规格（如330ml/500ml）、加入购物车或立即购买。

**实施分析**：详情页需显示当前用户所在门店的具体售价和库存（多店版下，同一商品在不同门店价格、库存可独立）。支持多规格SKU（如口味、重量）。集成 **旺道多维SKU引擎** ，可处理上万级SKU组合，并实时扣减库存。

**实现技术/方法**：商品主表spu，规格表sku。门店商品关联表store_spu存储门店价格、门店库存。加入购物车时调用Redis原子递减预占库存（防止超卖）。详情页使用骨架屏提升感知性能。

**算法与数据流**：
- 查询详情：根据spu_id和当前门店store_id，查询store_spu获得价格、库存；查询sku表展示规格列表。
- 规格选择：用户选择规格（如“500ml”）→ 前端显示对应sku的库存、价格 → 加入购物车写入Redis购物车hash。

**操作流程**：点击商品 → 查看多图轮播 → 选择“500ml”规格 → 点击“加入购物车” → 提示成功。

**FAQ**：
- Q：不同门店同一商品价格不同，用户会不会困惑？A：小程序顶部固定显示当前门店名称，用户可手动切换门店，价格明确。
- Q：加入购物车后商品降价了怎么办？A：结算时重新获取最新价格，以提交订单时刻为准。

### 6.4 购物车与结算

**应用场景**：用户将多个商品加入购物车，统一修改数量、删除，然后选择合适的门店、配送方式、优惠券，提交订单。

**实施分析**：购物车数据存储在Redis（用户ID为key），结构为hash，field为商品sku_id+门店ID，value为数量。结算时后端重新校验每个商品的最新价格、库存、配送范围。

**实现技术/方法**：Redis购物车有效期30天。提交订单时开启数据库事务：扣减门店库存（update ... where stock>= quantity）、生成订单、清空购物车对应项。若库存不足则回滚并提示。

**算法与数据流**：
- 结算页请求：后端遍历购物车商品 → 查询门店商品当前价格、库存 → 计算总金额 → 查询用户可用优惠券（未使用、未过期、满足门槛）→ 返回给前端。
- 下单：前端提交订单请求 → 后端开启事务 → 锁定库存行（select for update）→ 扣减 → 生成订单号（雪花算法）→ 插入订单表 → 清空购物车对应项 → 提交事务。

**操作流程**：购物车列表勾选商品 → 点击“去结算” → 选择收货地址（首次需添加）→ 选择“立即配送”并预约时间段 → 系统自动匹配门店 → 使用优惠券 → 提交订单。

**FAQ**：
- Q：购物车商品在不同门店怎么办？A：系统会按门店拆分订单，一次付款生成多个门店子订单，分别配送。
- Q：用户提交订单后未支付，库存是否锁定？A：锁定15分钟，超时未支付自动释放库存（定时任务扫描待支付订单）。

### 6.5 订单与支付

**应用场景**：用户提交订单后跳转支付页，调用微信支付。支付成功后可在“我的订单”查看所有状态（待配送、配送中、已完成、已取消），并申请退款（仅限未配送订单）。

**实施分析**：支付成功后异步回调通知后端更新订单状态。退款支持原路退回，需管理员审核。

**实现技术/方法**：微信支付V3接口，生成预支付订单。退款使用API证书。订单状态机设计严谨，防止状态跳跃。

**算法与数据流**：
- 支付流程：前端获取prepay_id → 调用wx.requestPayment → 支付成功 → 微信异步通知后端 /pay/notify → 后端签名验证 → 更新订单状态为“待配送” → 发送模板消息给用户。
- 退款流程：用户申请 → 管理员后台调用微信退款API → 微信异步通知退款结果 → 更新订单状态为“已退款”。

**操作流程**：提交订单 → 微信支付密码/指纹 → 成功 → 收到微信服务通知 → 订单列表显示“待配送”。

**FAQ**：
- Q：支付成功但订单状态未更新怎么办？A：异步通知可能延迟，系统提供手动同步按钮（主动查询微信支付状态）。
- Q：退款需要多久到账？A：微信支付原路退回通常1-3个工作日。

### 6.6 会员中心

**应用场景**：用户查看会员等级、积分、优惠券、收货地址、历史订单、积分商城兑换等。

**实施分析**：会员等级基于累计消费金额自动升级（如V1:0-499元，95折；V2:500-999元，9折；V3:1000元以上，85折）。积分规则：消费1元=1积分，积分可用于兑换商品或抵扣（100积分=1元）。

**实现技术/方法**：用户表member字段level、total_amount。每次订单完成后累加total_amount并重新计算等级（触发事件）。积分明细记录表。

**算法与数据流**：
- 等级计算：order_complete事件 → 更新用户total_amount → 根据配置的等级阈值对比 → 更新level，推送模板消息“恭喜升级”。
- 积分抵扣：结算页用户选择“使用积分” → 后端计算最高可抵扣金额（min(积分/100, 订单金额*0.3)）→ 订单表记录抵扣积分。

**操作流程**：点击“我的” → 显示会员等级V2 → 积分余额560 → 点击“积分商城”→ 兑换一包抽纸（需200积分）→ 确认兑换 → 生成兑换券码。

**FAQ**：
- Q：不同门店消费积分是否合并？A：所有门店消费均计入同一用户账户，积分通用。
- Q：积分会过期吗？A：后台可设置积分有效期（如自然年年底过期），过期前15天发送提醒。

### 6.7 营销活动参与（秒杀/拼团/优惠券）

**应用场景**：用户在首页看到秒杀倒计时，点击参与；或发起拼团分享给好友；领取优惠券。

**实施分析**：秒杀采用独立库存，与常规库存隔离。拼团需要单独订单表和拼团组表。优惠券支持领取、发放。

**实现技术/方法**：秒杀使用Redis预热库存，lua脚本原子扣减。拼团使用状态机，超时未成团自动退款。优惠券发放支持“新用户注册”“满额赠券”等规则。

**算法与数据流**：
- 秒杀下单：用户发起秒杀请求 → Redis lua扣减秒杀库存 → 成功则异步生成订单（扣减商品秒杀库存）→ 否则返回“已抢完”。
- 拼团：用户开团 → 生成拼团组（有效期24小时）→ 支付成功 → 等待其他人参团 → 参团后判断人数是否满团 → 满团则所有成员订单状态变更为“待配送”；超时未满，自动退款。

**操作流程**（拼团）：用户A点击“2人团” → 支付39.9元 → 分享给微信好友B → B进入页面参团并支付 → 两人均获得商品。

**FAQ**：
- Q：秒杀商品可以和普通商品一起下单吗？A：为简化库存逻辑，秒杀商品需单独下单，不支持合并购物车。
- Q：拼团失败退款是否扣除优惠券？A：退款时退还优惠券（若未使用）或按比例抵扣金额。

## 七、后台功能（PC管理端）

### 7.1 门店管理（超级管理员）

**应用场景**：总部创建分店（东城店、南城店），设置每家店的配送半径、营业时间、门店管理员账号。

**实施分析**：门店表store包含名称、地址、经纬度、配送半径(米)、营业开始/结束时间、状态（启用/禁用）。管理员账号关联门店ID。

**实现技术/方法**：使用PostGIS存储经纬度，计算用户距离。后台地图选点（腾讯地图API）获取坐标。利用 **旺道多角色安全引擎** 实现RBAC粒度控制，防止越权操作。

**数据流**：超级管理员添加门店 → 写入stores表 → 生成该门店独立的商品管理权限。

**操作流程**：超级管理员登录后台 → 点击“门店管理” → 新增门店 → 填写信息，地图标记位置 → 保存。

**FAQ**：
- Q：一个门店可以有多个店长吗？A：可以，分配多个管理员账号绑定同一门店ID。
- Q：配送半径修改后对已有订单有影响吗？A：不影响历史订单，新订单按新半径计算。

### 7.2 商品管理（门店店长+超级管理员）

**应用场景**：店长管理本店商品：上架、下架、调整价格和库存、设置秒杀库存、批量导入（扫码枪）。

**实施分析**：支持单个录入和批量导入。扫码枪扫描商品69码，系统自动从中央商品库匹配商品信息（名称、图片、类目），若匹配不到则弹出新增表单。支持库存预警（低于阈值高亮）。

**实现技术/方法**：中央商品库（product_base表）存储标准信息，门店商品表store_product存储门店价格、库存、上架状态。扫码枪输入焦点模拟键盘事件，前端监听回车触发查询。服务端利用 **旺道数据Agent** 自动同步商品基础信息变更至所有门店。

**算法与数据流**：
- 扫码快速入库：扫描69码 → 后端查询product_base → 存在则返回商品信息，店长仅需输入门店售价和库存 → 写入store_product；不存在则进入新建基础信息流程。
- 库存预警：定时任务（每小时）扫描store_product.stock < alert_threshold → 发送站内信通知店长。

**操作流程**：店长登录后台 → 商品管理 → 点击“新增商品” → 使用扫码枪扫描商品条码 → 系统自动填充名称、类目 → 录入门店价格99元、库存100件 → 保存。

**FAQ**：
- Q：如果总部统一调整某商品价格，如何同步所有门店？A：超级管理员在商品基础信息中修改“建议零售价”，各门店可手动选择“同步总价”。
- Q：商品图片需要多大？A：建议宽750px，比例1:1，系统自动压缩。

### 7.3 订单处理与配送指派

**应用场景**：门店收到新订单，店长查看订单详情、打印小票，指派配送员（或开启自动派单）。配送员端接收任务，更新配送状态。

**实施分析**：订单列表按状态分组（待配送、配送中、已完成）。指派支持手动选择和基于距离、空闲状态的智能推荐。配送员端独立小程序或H5。

**实现技术/方法**：WebSocket推送新订单提醒。智能派单使用贪心算法：计算每个可用配送员到门店距离 + 当前任务数加权。配送员完成送达拍照上传到OSS。

**算法与数据流**：
- 智能派单：新订单产生 → 查询该门店在线配送员列表 → 对每个配送员计算 score = (distance_km*2) + (current_orders*5) → 选择score最低者 → 推送订单。
- 配送轨迹：配送员点击“开始配送”后每30秒上报一次GPS，存储到位置历史表（用于调度监控）。

**操作流程**：店长登录后台 → 订单列表显示新订单（红色角标）→ 点击详情 → 打印小票 → 点击“指派配送员”→ 选择小张 → 小张登录配送端收到任务 → 扫码取货 → 开始配送 → 送达拍照 → 订单状态变“已完成”。

**FAQ**：
- Q：如果配送员拒单怎么办？A：店长可以重新指派，或标记为“无人接单”并短信提醒用户自提。
- Q：如何防止配送员虚假签收？A：要求拍照上传+地理位置校验（送达时GPS必须在收货地址100米内）。

### 7.4 营销活动管理（超级管理员）

**应用场景**：总部创建优惠券、秒杀、拼团、满减、积分兑换活动。

**实施分析**：活动可指定适用门店（全部或部分）。秒杀和拼团需设置活动库存、限购数量。优惠券支持“满100减10”等规则。

**实现技术/方法**：活动配置表存储规则（JSON格式）。定时任务提前将秒杀商品库存预热到Redis。拼团超时关闭使用延迟队列（RabbitMQ或Redis过期监听）。

**算法与数据流**：
- 秒杀活动开始：后台设置活动起止时间 → 提前30分钟将库存加载到Redis key:seckill:{activity_id}:{sku_id} → 用户抢购时lua扣减 → 活动结束后删除key。
- 满减自动匹配：结算时后端遍历所有启用满减活动，计算最优组合（动态规划，但门店级活动数量有限，线性匹配即可）。

**操作流程**：管理员登录后台 → 营销中心 → 新建秒杀 → 选择“进口车厘子”门店定价9.9元，活动库存50，限购1 → 设置时间20:00-20:30 → 发布 → 小程序首页倒计时显示。

**FAQ**：
- Q：秒杀可以和其他优惠叠加吗？A：秒杀商品默认不参加其他满减和优惠券，避免超低价亏损。
- Q：拼团未成团自动退款，会扣除手续费吗？A：微信支付会全额原路退回，企业无需承担手续费（已支付的手续费会返还）。

### 7.5 数据看板与报表

**应用场景**：总部查看全部门店销售趋势、商品排行、用户分析；店长查看本店报表。

**实施分析**：看板展示昨日销售额、环比增长、今日实时订单量。图表使用ECharts生成。支持导出Excel。

**实现技术/方法**：离线统计：每日凌晨2点定时任务聚合订单数据（order_detail + order）写入报表汇总表，看板直接查询汇总表，避免实时聚合压力。数据可导出采用NPOI组件。利用 **旺道数据Agent** 自动生成周报并邮件发送给店长。

**算法与数据流**：
- 销售额环比：计算(今日-昨日)/昨日。
- 商品销量排行榜：从order_detail表group by sku_id，取最近7天sum(quantity) top10。

**操作流程**：超级管理员进入“数据看板” → 看到各门店销售额柱状图 → 点击“导出月报表”→ 下载Excel。

**FAQ**：
- Q：数据看板可以实时吗？A：今日数据为实时查询（订单表），历史数据来自汇总表，误差不超过5分钟。
- Q：能否看到每个用户的复购率？A：支持，在“用户分析”模块提供复购率漏斗（首次下单 → 二次下单比例）。

### 7.6 权限与安全管理

**应用场景**：超级管理员创建运营、财务等角色，分配菜单权限；设置登录IP白名单、双因子认证（可选）。

**实施分析**：RBAC模型：用户→角色→权限。权限粒度到按钮级（如“导出报表”权限）。安全方面集成 **旺道加密引擎** 对敏感数据（手机号、地址）数据库加密存储。

**实现技术/方法**：JWT token鉴权 + 刷新token。加密使用AES-256。登录日志记录IP、设备。

**操作流程**：超级管理员 → 角色管理 → 新增“财务角色”→ 勾选“查看报表”权限 → 将财务人员账号分配该角色。

**FAQ**：
- Q：店长误删商品怎么办？A：系统有回收站功能，超级管理员可恢复。
- Q：如何防止员工泄露数据？A：后台可开启水印（用户名+时间），所有导出Excel带水印。

## 八、安全策略

- **传输安全**：全站启用SSL/TLS 1.3，强制HTTPS，防中间人攻击。SSL证书由环企提供代办并自动续期。
- **存储安全**：用户密码bcrypt加盐哈希；手机号、地址使用AES-256加密，加密密钥存储在独立密钥管理服务（KMS）。旺道加密引擎支持字段级动态脱敏，后台操作人员仅能看到前后3位。
- **应用安全**：防SQL注入（参数化查询）、XSS（输出过滤）、CSRF（token校验）。接口限流（同一IP每秒最多10请求）。旺道多角色安全引擎实现了细粒度访问控制，并自动拦截异常登录行为（异地登录、暴力破解）。
- **备份与灾备**：每日凌晨全量备份数据库与图片文件，保留最近30天；支持一键回滚。云服务器提供99.95%可用性承诺。
- **备案与合规**：代办小程序备案、网站备案，确保符合《个人信息保护法》，用户隐私政策弹窗告知。

## 十、注意事项

1. **数据初始化**：上线前需完成商品信息录入、门店信息配置、配送员注册。建议先完成100个高频商品的图文录入，环企提供免费录入服务。
2. **用户隐私**：小程序需配置隐私保护指引，明确收集位置、手机号用途。用户首次进入必须授权。
3. **支付对接**：需提前申请微信支付商户号，并配置好回调域名。退款需保证账户余额充足。
4. **配送范围测试**：系统根据直线距离判断配送范围，实际步行路线可能有差异，建议各门店实际测试后微调半径。
5. **售后规则**：明确生鲜商品不支持无理由退货，其他商品在未配送前可申请退款；已配送商品需用户拒收后门店核实再退款。
6. **高并发场景**：秒杀活动前建议临时扩容服务器配置（2核4G升4核8G），活动结束后降配。
7. **系统维护**：每月至少更新一次SSL证书（环企提供自动提醒）；定期清理30天前的缓存日志。
